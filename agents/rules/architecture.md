# Architecture Rules

| ID  | Rule                                                                                                                                                                                                                                | Why                     |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------- |
| A01 | **Single responsibility**: each module has one clear purpose                                                                                                                                                                        | Maintainability         |
| A02 | **Module isolation**: each module or subsystem has clear ownership boundary. Directory structure should follow runtime/analysis boundaries, not arbitrary decomposition. Apply only where module separation is genuinely justified. | Independent testing     |
| A03 | **Pure core logic**: no side effects, no I/O in analysis functions. Source code must be prepared by the I/O layer (bootstrap, CLI entry points, I/O adapters) and passed into pure analysis functions as AST or abstract models (Data-driven analysis pattern). I/O is only permitted in `src/cli/` and dedicated adapter modules. | Deterministic, testable |
| A04 | **Dependency injection**: constructor/args, no globals                                                                                                                                                                              | Mockability             |
| A05 | **No circular dependencies**: unidirectional data flow. Enforce automatically via `madge` or `eslint-plugin-import/no-cycle` in CI.                                                                                                  | Clear architecture      |
| A06 | **Error handling strategy**: use typed exceptions for unexpected core failures; use `Result`-style returns for expected/recoverable failure paths. Keep error semantics consistent across layers.                                     | Predictable error flow  |
