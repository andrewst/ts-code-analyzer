# RFC: Утилита обнаружения мёртвого кода в библиотеках TypeScript

## 1. Обзор

Данный документ описывает архитектурное решение утилиты статического анализа для обнаружения мёртвого кода в экспортах библиотек TypeScript. Утилита выявляет экспортируемые символы, которые не потребляются публичным API библиотеки, юнит-тестами или демонстрационным кодом.

## 2. Постановка проблемы

Большие библиотеки TypeScript со временем накапливают экспортируемые символы без чёткого понимания, какие экспорты действительно используются. Экспорты, существующие только во внутренних файлах, но не переэкспортируемые через точки входа публичного API, или используемые только в тестах/демо-коде, представляют собой мёртвый код, увеличивающий затраты на поддержку и размер бандла.

## 3. Основные понятия

### 3.1 Определение мёртвого кода

Экспортируемый символ считается **мёртвым кодом**, когда:

- Он экспортируется из TypeScript-файла внутри библиотеки
- Он НЕ переэкспортируется через какую-либо точку входа публичного API (файлы в корне `src/`)
- Он НЕ используется в рабочем коде библиотеки (исключая файл, где он определён)
- Использование ТОЛЬКО в тестах (`./tests`) или демо (`./demo`) НЕ считается валидным использованием

### 3.2 Определение публичного API

Публичный API состоит из:

- Всех файлов в корневом каталоге `src/`, содержащих экспорты (например, `src/index.ts`, `src/utils.ts`)
- Всех символов, транзитивно экспортируемых через эти точки входа
- Цепочки реэкспортов: если файл A экспортирует из файла B, а файл B экспортирует из файла C, все три символа являются частью публичного API

### 3.3 Определение валидного использования

Символ имеет валидное использование, когда:

- Он потребляется в исходном коде библиотеки (не только в тестах/демо)
- Он является частью публичного API (напрямую или транзитивно)
- Внутреннее использование в том же файле НЕ считается валидным использованием

## 4. Архитектура системы

### 4.1 Компоненты верхнего уровня

Утилита состоит из пяти основных компонентов:

1. **Сканер проекта**: Обнаруживает и каталогизирует все TypeScript-файлы в библиотеке
2. **Парсер и анализатор AST**: Парсит TypeScript-файлы и извлекает связи экспорт/импорт с использованием анализа абстрактного синтаксического дерева
3. **Построитель публичного API**: Формирует полную поверхность публичного API, анализируя файлы точек входа и их цепочки реэкспортов
4. **Построитель графа использования**: Строит ориентированный граф использования символов во всей кодовой базе
5. **Детектор мёртвого кода**: Сравнивает экспортируемые символы с паттернами использования для выявления мёртвого кода

### 4.2 Ответственность компонентов

#### 4.2.1 Сканер проекта

**Ответственность:**

- Рекурсивное сканирование каталога `src/` библиотеки для всех файлов `.ts` и `.tsx`
- Сканирование каталога `tests/` для файлов тестов
- Сканирование каталога `demo/` для демонстрационных файлов
- Идентификация точек входа публичного API (все файлы, содержащие экспорты, в корне `src/`)
- Нормализация путей файлов и обработка алиасов путей из `tsconfig.json`

**Результат:**

- Полный инвентарь файлов, категоризированных по типу (исходный код библиотеки, тесты, демо, точки входа публичного API)
- Параметры конфигурации TypeScript (paths, baseUrl и т.д.)

#### 4.2.2 Парсер и анализатор AST

**Ответственность:**

- Парсинг каждого TypeScript-файла в абстрактное синтаксическое дерево
- Извлечение всех объявлений экспорта (именованные экспорты, экспорты по умолчанию, реэкспорты)
- Извлечение всех объявлений импорта и их паттернов использования
- Обработка различных форм экспорта:
  - `export const/function/class/interface/type`
  - `export { name }`
  - `export { name } from './path'`
  - `export * from './path'`
  - `export default`
- Разрешение ссылок на символы до их определений
- Отслеживание, какие импортированные символы действительно используются в теле файла

**Результат:**

- Манифест экспортов для каждого файла со всеми экспортированными символами и их типами
- Манифест импортов для каждого файла со всеми импортированными символами и их статусом использования
- Расположения определений символов для межфайлового разрешения

#### 4.2.3 Построитель публичного API

**Ответственность:**

- Начало от идентифицированных точек входа публичного API
- Рекурсивное отслеживание всех цепочек реэкспортов
- Формирование полного набора символов, образующих публичный API
- Обработка циклических реэкспортов и обнаружение бесконечных циклов
- Разрешение алиасированных экспортов (например, `export { foo as bar } from './path'`)
- Отслеживание файла-источника для каждого публично экспортируемого символа

**Алгоритм:**

1. Сбор всех файлов точек входа из корня `src/`
2. Для каждой точки входа — парсинг экспортов
3. Для реэкспортов (`export ... from`) — рекурсивная обработка исходного файла
4. Построение маппинга: имя символа → файл-источник → является частью публичного API
5. Продолжение до полного разрешения всех цепочек реэкспортов

**Результат:**

- Полный набор символов публичного API с маппингами файлов-источников
- Граф цепочек реэкспортов для отладки/визуализации

#### 4.2.4 Построитель графа использования

**Ответственность:**

- Построение ориентированного графа всего использования символов в кодовой базе
- Узлы: символы (функции, классы, константы, типы, интерфейсы)
- Рёбра: связи использования (файл A использует символ из файла B)
- Категоризация рёбер по контексту (исходный код библиотеки, тесты, демо)
- Отслеживание межфайловых зависимостей

**Построение графа:**

1. Для каждого файла — идентификация всех импортированных символов
2. Для каждого импорта — проверка, используется ли символ в теле файла
3. Создание рёбер от места использования к определению символа
4. Метка рёбер контекстом использования (source/test/demo)
5. Исключение самоссылок (символ, используемый в своём собственном файле определения)

**Результат:**

- Полный граф использования с рёбрами, размеченными по контексту
- Обнаружение несоответствий импорта/экспорта (импортировано, но локально не используется)

#### 4.2.5 Детектор мёртвого кода

**Ответственность:**

- Запрос к графу использования и набору публичного API для выявления мёртвого кода
- Применение правил обнаружения мёртвого кода
- Генерация отчётов обнаружения с actionable-информацией

**Правила обнаружения:**

Символ является **мёртвым кодом**, если выполняется ЛЮБОЕ из этих условий:

- Символ экспортирован, но НЕ в публичном API И НЕ используется в исходном коде библиотеки
- Символ экспортирован и находится в публичном API, НО не имеет внешних потребителей (крайний случай для недавно добавленных экспортов)
- Символ используется ТОЛЬКО в тестах или демо (независимо от статуса в публичном API)

Символ является **валидным**, если:

- Символ находится в публичном API И используется хотя бы в одном файле, не являющемся тестом/демо
- Символ используется в исходном коде библиотеки другими файлами (внутренняя утилита)

**Результат:**

- Список символов мёртвого кода с метаданными
- Классификация причин, почему каждый символ считается мёртвым
- Предложения по исправлению

```
       ↓

       ↓
       ├─────────────────┬──────────────────┐

```

## 6. Стратегия реализации

### 6.1 Технологический стек

**Рекомендуемый подход:**

- Использовать TypeScript Compiler API или специализированную библиотеку парсинга AST (например, `ts-morph`, `@typescript-eslint/typescript-estree`)
- Эти библиотеки предоставляют высокоуровневые API для:
  - Парсинга TypeScript-файлов с полной информацией о типах
  - Обхода узлов AST
  - Разрешения ссылок на символы
  - Обработки конструкций, специфичных для TypeScript (дженерики, неймспейсы, модули)

**Альтернативный подход:**

- Использовать более простой текстовый поиск по паттернам с regex (не рекомендуется из-за сложности и ненадёжности)

### 6.2 Фаза 1: Сканирование проекта и обнаружение файлов

**Шаги:**

1. Чтение `tsconfig.json` для понимания структуры проекта
2. Разрешение алиасов путей и параметров компилятора
3. Обход каталогов `src/`, `tests/`, `demo/`
4. Категоризация файлов по типу и расположению
5. Идентификация точек входа публичного API (файлы в корне `src/` с экспортами)

**Важные моменты:**

- Обработка паттернов `exclude` и `include` из `tsconfig.json`
- Поддержка структур монорепозиториев при необходимости
- Корректная обработка файлов объявлений `.d.ts`

### 6.3 Фаза 2: Парсинг AST и извлечение символов

**Шаги:**

1. Парсинг каждого TypeScript-файла с помощью выбранного парсера
2. Извлечение всех объявлений экспорта с метаданными:
   - Имя символа
   - Тип символа (функция, класс, константа, интерфейс, тип и т.д.)
   - Вид экспорта (именованный, по умолчанию, реэкспорт)
   - Расположение в источнике (путь файла, номер строки)
3. Извлечение всех объявлений импорта:
   - Имена импортированных символов
   - Исходный модуль
   - Вид импорта (именованный, по умолчанию, неймспейс)
4. Анализ тела файла для определения, какие импортированные символы действительно используются
5. Построение таблиц символов для каждого файла

**Краевые случаи для обработки:**

- Динамические импорты (`import()`)
- Неймспейс-импорты (`import * as ns from './path'`)
- Неймспейс-экспорты (`export as namespace ns`)
- Ambient-объявления
- Слияние объявлений
- Дженерик-типы и алиасы типов
- Импорт только типов (`import type { ... }`)

### 6.4 Фаза 3: Построение публичного API

**Шаги:**

1. Начало с файлов точек входа в корне `src/`
2. Для каждой точки входа:
   - Парсинг всех экспортов
   - Для реэкспортов — рекурсивная обработка исходных файлов
   - Отслеживание цепочек реэкспортов для избежания бесконечных циклов
3. Построение полного набора символов публичного API
4. Маппинг каждого публичного символа к его файлу-источнику

**Ключевые моменты алгоритма:**

- Обнаружение циклов в цепочках реэкспортов (использовать множество посещённых)
- Корректная обработка алиасированных экспортов
- Различие между `export * from` (barrel) и `export { name } from` (селективный)
- Отслеживание, какие символы действительно публичные, а какие — внутренние утилиты, проникающие через экспорты

### 6.5 Фаза 4: Построение графа использования

**Шаги:**

1. Для каждого файла в кодовой базе:
   - Получение всех импортированных символов
   - Проверка, какие импортированные символы действительно используются в теле файла
   - Создание ориентированных рёбер от места использования к определению символа
2. Метка каждого ребра контекстом (source/test/demo)
3. Построение представления графа в виде списка смежности

**Логика обнаружения использования:**

- Для каждого импорта — поиск в AST файла ссылок на импортированное имя
- Обработка деструктуризации (`const { foo } = bar`)
- Обработка доступа к свойствам (`obj.method()`)
- Обработка ссылок на типы (при анализе типов, а не только значений)

### 6.6 Фаза 5: Обнаружение мёртвого кода

**Шаги:**

1. Для каждого экспортированного символа в кодовой базе:
   - Проверка, находится ли символ в наборе публичного API
   - Проверка, имеет ли символ рёбра использования в исходном коде библиотеки
   - Проверка, имеет ли символ ТОЛЬКО рёбра использования в тестах/демо
2. Применение правил обнаружения:
   - Если НЕ в публичном API И НЕ используется в исходном коде → мёртвый код
   - Если используется ТОЛЬКО в тестах/демо → мёртвый код
   - В противном случае → валидный код
3. Генерация отчёта с классификациями

**Поля отчёта для каждого символа:**

- Имя символа
- Тип символа (функция, класс, константа и т.д.)
- Путь файла, где определён
- Тип экспорта (именованный, по умолчанию, реэкспорт)
- Причина классификации (почему это мёртвый код или валидный)
- Список файлов, где он используется (для контекста)
- Номер строки в исходном файле

Утилита должна поддерживать конфигурацию через:

### 8.1 Аргументы CLI

```bash
detect-dead-code \
  --src ./src \
  --tests ./tests \
  --demo ./demo \
  --tsconfig ./tsconfig.json \
  --format json \
  --output report.json \
  --fail-on-dead-code
```

Поддержка файла `.deadcoderc.json` или аналогичного:

```json
{
  "src": "./src",
  "tests": "./tests",
  "demo": "./demo",
  "tsconfig": "./tsconfig.json",
  "exclude": ["**/*.spec.ts", "**/*.test.ts"],
  "ignoreSymbols": ["someKnownUtility"],
  "failOnDeadCode": true,
  "reportFormat": "json"
}
```

## 9. Краевые случаи и ограничения

### 9.1 Известные ограничения

1. **Динамические экспорты/импорты**: Невозможно обнаружить динамически генерируемые экспорты или импорты
2. **Использование на основе рефлексии**: Невозможно обнаружить использование через рефлексию или инспектирование во время выполнения
3. **Использование только типов**: Может некорректно идентифицировать импорт/экспорт только типов, если они не размечены явно
4. **Импорты побочных эффектов**: Невозможно определить, необходимы ли импорты только побочных эффектов (`import './polyfills'`)
5. **Точки расширения/плагинов**: Символы, предназначенные как точки расширения, могут казаться неиспользуемыми
6. **Использование третьими сторонами**: Невозможно проанализировать, как внешние потребители используют библиотеку (опирается на предположение о границе публичного API)

### 9.2 Краевые случаи

1. **Циклы реэкспортов**: Должен обрабатывать циклы A → B → A без бесконечных циклов
2. **Неймспейс-экспорты**: `export * as ns from './path'` требует специальной обработки
3. **Слияние объявлений**: Символы с одинаковым именем в разных контекстах
4. **Условные экспорты**: Экспорты, ограниченные type guards или условными типами
5. **Ambient-объявления**: Объявления `declare` в файлах `.d.ts`
6. **Параметры дженерик-типов**: Параметры типов, которые кажутся неиспользуемыми в сигнатурах функций
7. **Barrel-файлы**: Файлы, которые только переэкспортируют из других файлов
8. **Index-файлы**: Файлы `index.ts` в подкаталогах (не часть публичного API по текущей спецификации)

## 10. Стратегия тестирования

### 10.1 Юнит-тесты

- Тестирование каждого компонента изолированно с мокированными данными
- Парсер AST: проверка корректности извлечения экспортов/импортов из различных конструкций TypeScript
- Построитель публичного API: проверка корректности разрешения цепочек реэкспортов
- Построитель графа использования: проверка корректности построения графа
- Детектор мёртвого кода: проверка корректности классификации по известным сценариям

### 10.2 Интеграционные тесты

## 11. Производительность

### 11.1 Масштабируемость

- Парсинг AST ресурсоёмок для больших кодовых баз
- Рассмотреть параллельный парсинг независимых файлов
- Кэшировать результаты парсинга AST между запусками, если файлы не изменились
- Использовать инкрементальный анализ (повторно парсить только изменённые файлы)

### 11.2 Использование памяти

- Большие кодовые базы могут потреблять значительную память для хранения AST
- Рассмотреть потоковую/итеративную обработку для очень больших библиотек
- Освобождать AST после извлечения для высвобождения памяти

### 11.3 Ожидаемая производительность

- Маленькая библиотека (< 100 файлов): < 5 секунд
- Средняя библиотека (100-500 файлов): < 30 секунд
- Большая библиотека (500+ файлов): < 2 минут

## 15. Критерии успеха

Утилита считается успешной, когда:

1. Корректно идентифицирует > 95% мёртвого кода в тестовых фикстурах
2. Выдаёт < 5% ложных срабатываний
3. Завершает анализ кодовой базы из 500 файлов < 2 минут
4. Предоставляет actionable отчёты, снижающие затраты на поддержку библиотеки
5. Гладко интегрируется в CI/CD конвейеры
6. Принимается мейнтейнерами библиотек и показывает измеримое улучшение качества кода

## 16. Глоссарий

- **Мёртвый код**: Экспортируемые символы, не потребляемые публичным API или исходным кодом библиотеки
- **Публичный API**: Символы, экспортируемые из файлов точек входа в корневом каталоге `src/`
- **Реэкспорт**: Экспорт, перенаправляющий символы из другого модуля
- **Точка входа**: Файл в корне `src/`, экспортирующий символы для публичного потребления
- **Граф использования**: Ориентированный граф отношений использования символов между файлами
- **AST**: Abstract Syntax Tree, парсенное представление исходного кода
- **Символ**: Любая именованная сущность (функция, класс, константа, тип, интерфейс и т.д.)
- **Barrel-файл**: Файл, который только переэкспортирует из других модулей без добавления новых экспортов
- **Контекст**: Является ли файл исходным кодом библиотеки, тестом или демо-кодом
