# Описание задачи

## Краткое описание

Когда TypeScript (npm) библиотека вырастает до больших размеров, контролировать используемый код становится очень сложно.
При экспорте объектов из файла неочевидно, будут ли они использоваться в публичном API или нет.
Если экспортируемые сущности не используются внутри библиотеки, в тестах или в демо-примерах, они могут долго оставаться в коде как мёртвый груз.

## Цель

Спроектировать утилиту для обнаружения мёртвого кода среди экспортируемых объектов и функций TypeScript библиотеки.

Утилита должна выявлять:

- Файлы с экспортируемыми сущностями, которые не используются в публичном API библиотеки;
- Конкретные экспортируемые объекты/функции, не имеющие использований в публичном API.

**Важно:** Если функция используется только в тестах или в демо, но не используется в публичном API и внутри самой библиотеки, она считается мёртвым кодом.

## Входные данные

1. Исходный код TypeScript библиотеки;
2. Публичный API определяется файлами с экспортами в корне директории `src/` (не только `index.ts`, но и другие файлы, например `src/utils.ts`);
3. Внешние потребители библиотеки могут использовать только экспорты из публичного API (контроль запрета на использование внутреннего кода в обход публичного API осуществляется другой утилитой);
4. Юнит-тесты расположены в директории `./tests` в корне репозитория;
5. Демо-пример использования библиотеки находится в каталоге `./demo`;
6. Конфигурация TypeScript (`tsconfig.json`) для корректного разрешения путей и зависимостей.

## Выходные данные

Утилита должна предоставить:

1. Список файлов, содержащих мёртвый код (экспорты без использований);
2. Для каждого файла — перечень конкретных экспортируемых сущностей (функций, классов, интерфейсов, типов, констант), которые не используются;
3. Формат вывода: консольный отчет (text/json) с возможностью экспорта в файл.

## Критерии мёртвого кода

Экспортируемая сущность считается **мёртвым кодом**, если выполняется **все** следующее:

- Не импортируется и не используется в других файлах библиотеки (за исключением тестов и демо);
- Не входит в публичный API библиотеки (не реэкспортируется из файлов в корне `src/`);
- Используется **только** в тестах (`./tests`) или демо (`./demo`), либо не используется вовсе.

## Технические требования

- Анализ на уровне AST (Abstract Syntax Tree) для точного отслеживания импортов/экспортов;
- Поддержка TypeScript синтаксиса (включая type-only импорты/экспорты, namespaces, re-exports);
- Рекурсивное отслеживание зависимостей (если A экспортирует B, а B использует C, то C не считается мёртвым);
- Игнорирование сторонних зависимостей (node_modules).

## Ограничения

- Утилита не анализирует динамические импорты (`import()`);
- Не отслеживает рефлексию и метапрограммирование (например, доступ через `window['funcName']`);
- Не учитывает экспорты, которые могут использоваться только в runtime через сторонние инструменты (bundlers, плагины).
